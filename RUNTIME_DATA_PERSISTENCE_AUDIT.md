# Runtime Data Persistence Audit

Why the deployed app lost paid-ingestion data after a redeploy,
and the persistence design we landed to stop it from happening
again.

## Symptom

After a successful paid Week 1 ingestion + admin migration on
Railway, a new build/redeploy cleared:

- the canonical odds file at
  `data/processed/odds/2025/week-1-prop-markets.csv`
- the legacy flat files (`data/processed/prop_markets.csv`,
  `prop_quotes.csv`)
- the admin-state file at
  `data/admin/ingestion-state.json` — so `smokeSucceededAt` was
  reset and the paid Week 1 ingestion button re-locked
- the stored-backtest output files at
  `data/backtests/2025/week-1-*.fixture.json`
- the raw API usage log at `data/raw/api-usage/*.jsonl`

This is not data corruption — the container's filesystem is
ephemeral.

## Why

Railway's web-service containers do NOT have persistent
filesystems by default. Each build produces a new container
image; each redeploy starts a fresh container; restarts wipe
anything written under the working directory. **Nothing written
at runtime survives a redeploy unless it lives in a volume or
an attached database.**

The git-tracked schedule fixture
(`data/fixtures/nfl/2025-week-1-schedule.fixture.json`) survives
because it's baked into the image at build time. Same for
`data/backtests/2025/week-1-*.fixture.json` if those are
committed. The runtime-generated paid odds + admin state are
NOT committed (gitignored on purpose — they're paid + private),
so they vanish.

## Classification

| File | Generated by | Expensive to regenerate? | Persist? | Cache OK? |
|---|---|---|---|---|
| `data/processed/prop_markets.csv` | Paid Week 1 ingestion (566+ credits) | **Yes — costs money** | **DB + file** | yes |
| `data/processed/prop_quotes.csv` | Paid Week 1 ingestion | **Yes — costs money** | DB + file | yes |
| `data/processed/odds/2025/week-1-prop-markets.csv` | Admin migration from the two above | Cheap (file IO), but depends on the two above | **DB + file** | yes |
| `data/admin/ingestion-state.json` | Admin actions (smoke success, etc.) | Cheap, but losing it forces a re-run of paid smoke | **DB + file** | yes |
| `data/admin-ingestion/latest-*.json` | Admin actions (result snapshots) | Cheap, but loses audit trail | DB-optional + file | yes |
| `data/backtests/2025/week-1-*.fixture.json` | Stored backtest runner | Cheap (pure CPU once odds + nflverse are present) | **DB + file** | yes |
| `data/raw/api-usage/*.jsonl` | Paid ingestion (live mode) | Audit trail; can't be recreated | DB-optional + file | acceptable to lose if `ApiUsageLog` table is populated |
| `data/cache/odds-api/*.json` | HTTP cache for Odds API responses | Can be re-derived (still costs credits) | file only | yes |
| `data/raw/nfl/<season>/*.csv` | nflverse network fetch | Free to re-fetch | file only | yes |
| `data/processed/nfl/games.csv` + `player_week_stats.csv` + `rosters.csv` | nflverse normalizer | Free to re-fetch + re-normalize | file only | yes |

The expensive items — anything traceable to a real Odds API
call — get DB persistence + a file mirror. Cheap items keep the
file-only path; if they vanish, the admin action can be re-run
without spending a single credit.

## Persistence design

Railway already exposes `DATABASE_URL` (the existing Prisma
config provider is `postgresql`). Postgres is the durable store
when configured; the filesystem stays as a cache layer.

### New Prisma models

- `OddsIngestionRun` — per paid run. `(season, week, scope,
  status, startedAt, finishedAt, creditsEstimated, creditsUsed,
  creditsRemaining, marketsRequested, gamesRequested,
  errorMessage, rawResultJson)`. Audit trail of what was paid
  for.
- `StoredPropMarket` — canonical per-row odds. `(season, week,
  gameId, eventId, marketKey, playerName, playerId, team,
  opponent, propType, line, sportsbook, overOdds, underOdds,
  snapshotTime, oddsSource, isBeforeKickoff)`. Unique on
  `(season, week, marketKey, sportsbook, snapshotTime)` so
  re-running the migration is idempotent.
- `StoredBacktestRun` — full stored-mode output. `(season,
  week, dataMode, status, realWeek1BacktestReady,
  scheduleValidationStatus, syntheticFixture, candidatesJson,
  resultsJson, v1v2Json, parlayJson, gameEdgeJson)`. Indexed
  by `(season, week, createdAt)` so the page can fetch the
  latest run.
- `AdminIngestionState` — singleton (`id = "singleton"`).
  `(smokeSucceededAt, smokeCreditsUsed,
  week1IngestionSucceededAt, week1SubsetSucceededAt, lastAction,
  lastResultJson)`. Read at admin-page load; written after
  every admin action.

The pre-existing `ApiUsageLog` model is reused as the durable
form of the JSONL usage log — its table just needs to exist.

### Persistence layer

`src/lib/persistence/week-1-persistence.ts` exposes a thin
client interface with eight functions:

- `saveCanonicalOddsRowsToDb`, `loadCanonicalOddsRowsFromDb`
- `saveAdminIngestionStateToDb`, `loadAdminIngestionStateFromDb`
- `saveStoredBacktestRunToDb`, `loadLatestStoredBacktestRunFromDb`
- `saveOddsIngestionRunToDb`, `loadLatestOddsIngestionRunFromDb`

Every method returns `{ ok, … }` and never throws. When
`DATABASE_URL` is unset OR the Prisma import fails OR the
underlying call rejects, the result is `{ ok: false, error }`
and the caller falls back to the file cache. **No paid action
is ever blocked on a DB failure.**

For tests, a `PersistenceClient` interface lets the runner take
an in-memory stub; no real Postgres or even Prisma client
needed during CI.

### Load priority

1. **Stored odds**: canonical file → DB → legacy file →
   `MISSING_STORED_ODDS`.
2. **Admin state**: DB (singleton row) → file fallback → empty
   defaults. The DB wins when both exist; the file is rewritten
   on every state change so it stays close to the DB view for
   diagnostics.
3. **Stored backtest output**: DB (most recent
   `StoredBacktestRun` for `(season, week)`) → file → run
   fresh.

After loading from DB when the file is missing, the loader
**rehydrates the file cache** so future reads in the same
container are file-only. This keeps the hot path identical to
the pre-Postgres flow.

### Diagnostics

The admin page + diagnostics page now surface:
- `Persistent storage: Postgres available yes/no`
- `Canonical odds source: file / postgres / legacy / missing`
- `Admin ingestion state source: postgres / file / missing`
- `Stored backtest source: postgres / file / missing`
- An amber warning when the only source is the ephemeral
  filesystem, so the user sees the persistence risk before a
  redeploy hits.

`DATABASE_URL` and every other secret is never echoed back to
the client — only the boolean "configured" appears.

## Required deploy step

After the new models are pulled in, the Railway database needs
the corresponding tables. The project has no existing migration
history (no `prisma/migrations/`); the existing pattern is
`prisma db push`. Two paths:

1. **One-time admin sync.** SSH into the Railway service and
   run `npx prisma db push` against the production
   `DATABASE_URL`. Idempotent — re-running adds new columns
   without dropping data.

2. **Automated.** Add `prisma db push --skip-generate` to the
   `start` script so every container boot ensures the schema
   is current. Safe because `db push` only adds (doesn't drop)
   when the schema is purely additive.

Both options are covered in `RAILWAY_DEPLOYMENT.md`. **If the
table is missing, persistence quietly falls back to file only**
— the admin page surfaces this as "Persistent storage: no" so
the user notices before spending credits a second time.

## Standing rules (unchanged)

- Persistence layer never calls a paid API.
- DB writes never store `ODDS_API_KEY`, `ADMIN_INGEST_TOKEN`,
  raw API response bodies, or any other secret. Hashes /
  truncated previews / counts only.
- Migration helper is idempotent — re-running `db push` against
  the existing tables is safe.
- Local development without `DATABASE_URL` works exactly as
  before; the file-only path is unchanged.
- No model logic touched. Persistence sits at the IO boundary.
